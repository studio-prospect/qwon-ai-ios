# PREXUS App Icon Strategy

## Purpose

This document defines the visual direction for the PREXUS iPhone app icon.

The icon needs to feel:

- native to Apple platforms
- calm and precise
- recognizably AI-related without looking generic
- aligned with PREXUS as a runtime, not as a chat toy

## Product Fit

PREXUS is not a generic assistant app.

Its identity is built around:

- local-first intelligence
- routing and orchestration
- multimodal sensing
- persistent cognitive runtime behavior

Because of that, the icon should communicate:

- an intelligence core
- sensing and orchestration
- quiet confidence

It should avoid:

- obvious chat metaphors
- mascots
- cartoon robotics
- decorative futurism

## Apple Platform Constraints

The icon should follow Apple’s current app-icon direction:

- simple primary form
- minimal visual clutter
- recognizable at small sizes
- strong silhouette
- background that supports system-applied lighting and rendering

For PREXUS, that means:

- avoid over-detailed linework
- avoid text in the icon
- avoid tiny symbolic detail that only reads at large size
- prefer a clear central motif with a calm background

## Design Principles

### 1. Runtime-first, not chat-first

The icon should suggest a cognitive core, not a speech bubble.

### 2. Abstract, but legible

It should feel intelligent and technical without becoming cryptic.

### 3. Calm premium tone

PREXUS should feel serious, quiet, and Apple-native.

### 4. Strong at a glance

The icon must read clearly on the Home Screen and in Settings, search, and notifications.

## Color Direction

Recommended palette direction:

- deep graphite / near-black base
- cool silver or soft white structural highlight
- restrained electric blue or faint violet accent

This should create:

- contrast without loudness
- premium feel without looking game-like
- compatibility with future Liquid Glass-informed UI

Avoid:

- overly saturated neon
- rainbow gradients
- warm playful color systems
- bright green chatbot associations

## Concept Candidates

## Concept A — Core Orb

### Summary

A central luminous core surrounded by subtle concentric structure.

### Meaning

- local intelligence nucleus
- computation close to the device
- a contained but active runtime

### Visual structure

- dark background
- one bright central orb or node
- one or two soft structural rings or fields around it
- minimal glow, tightly controlled

### Strengths

- most aligned with PREXUS as a runtime
- highly recognizable
- scalable to small sizes
- not easily confused with a chat app

### Risks

- can become too generic if it looks like a plain glowing dot
- must avoid sci-fi wallpaper energy

## Concept B — Lens Aperture

### Summary

An abstract aperture / lens form representing sensing, OCR, vision, and routing.

### Meaning

- multimodal understanding
- sensor platform identity
- selective focus and interpretation

### Visual structure

- radial form with 4–6 simplified segments
- subtle depth or layered petal geometry
- strong central focal point

### Strengths

- ties well to vision + sensing
- can feel premium and precise
- works with Apple-like geometric reduction

### Risks

- can drift too close to camera-app symbolism
- may overemphasize vision relative to broader runtime identity

## Concept C — Node Field

### Summary

A central node with a shaped surrounding field suggesting routing, memory, and semantic flow.

### Meaning

- orchestration
- context field
- information shaping around a core

### Visual structure

- one main node
- one asymmetric or softly directional field
- subtle energy lines or layered halo geometry

### Strengths

- best expresses routing/orchestration specifically
- feels more unique than a simple orb
- avoids direct chat symbolism

### Risks

- harder to keep simple
- may lose clarity at smaller sizes if overdesigned

## Selected Direction

### Production choice: Glowing Converging Paths

After concept exploration, PREXUS should ship with a routing-first icon rather than a core-orb or lens metaphor.

Why:

- it expresses multiplexing, orchestration, and context compression directly
- it avoids surveillance, camera, or "watching eye" associations
- it reads as a calm signal-routing mark rather than a generic chatbot
- it fits PREXUS better as a persistent runtime that gathers many inputs and resolves them into one coherent path

### Visual structure

- several luminous paths enter from the left
- those paths softly converge into a single brighter route
- the background stays dark graphite to preserve contrast and calm
- glow stays blue-violet, cyan, and icy white rather than warm red

### Tone target

The icon should feel:

- quietly intelligent
- Apple-native
- softly luminous rather than metallic
- routing-aware rather than character-based

This means the final mark should borrow from Apple-style glow and layered light, but stop short of becoming ornamental.

## Concepts To Avoid

Avoid these directions unless there is a strong new branding reason:

### Speech bubble

Reason:

- makes PREXUS read as a generic chat app

### Brain icon

Reason:

- too literal
- dated AI metaphor

### Robot face

Reason:

- undermines calm, premium positioning
- pushes PREXUS toward novelty

### Lettermark-first icon

Reason:

- `P` or `PX` could work later, but at this stage it is more likely to feel like a generic SaaS brand than a runtime product

## Execution Guidance

When the icon moves to mockup or production:

1. start with grayscale silhouette tests
2. verify recognizability at small size
3. add restrained accent color only after the structure works
4. test against light and dark Home Screen contexts
5. keep the background simple enough for Apple’s system rendering

## Suggested Next Step

The next design step should be:

1. generate 3 visual icon mocks based on Concepts A, B, and C
2. compare them at Home Screen scale
3. choose one direction
4. refine toward production icon layers

## Decision Summary

PREXUS should use an icon that represents a **cognitive core** rather than a chat metaphor.

Current recommendation:

- **choose Core Orb as the first visual exploration**
- optionally evolve toward a **Core Orb + Node Field hybrid**
