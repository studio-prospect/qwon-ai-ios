# OpenAI

## Role in PREXUS

OpenAI is a Tier 3 cloud inference target in PREXUS. It should be used selectively for tasks where local inference is insufficient and where OpenAI-class models are a strong fit.

Typical target use cases:

- large code analysis
- structured reasoning
- long-context synthesis
- tool-oriented task decomposition
- high-quality text generation when local quality is insufficient

OpenAI should not be the default path for every request. PREXUS must route to it only after local analysis, privacy checks, and context compression.

## Product Positioning

Within PREXUS routing, OpenAI is generally a candidate for:

- code-heavy requests
- technical reasoning
- mixed structured and natural-language generation
- tasks that benefit from stronger general-purpose reasoning

This is a routing preference, not a hardcoded rule.

## Integration Principles

- Requests must pass through the PREXUS orchestrator
- Context must be compressed before transmission
- Provider-specific prompts should be generated by an adapter layer
- Raw full conversation history should not be forwarded by default
- Sensitive data must be filtered according to policy

## Required Capabilities

The OpenAI integration layer should support:

- authenticated request execution
- compact text generation over the Responses API
- structured prompt packaging
- provider-specific error handling
- retry and fallback signaling to the orchestrator

Future support may include:

- multimodal escalation paths
- structured outputs where beneficial
- batch or background workflows if product needs justify them

## Adapter Design

Suggested ownership in PREXUS:

- `runtime/llm/cloud/`: cloud execution flow
- `runtime/llm/adapters/`: OpenAI-specific request/response mapping
- `runtime/llm/prompts/`: reusable compact provider prompt templates

The adapter should convert a PREXUS route package into:

- provider-ready input
- system instruction block
- compressed context block
- optional memory block
- output normalization metadata

## Context Policy

Before sending a request to OpenAI, PREXUS should try:

- intent extraction
- history summarization
- duplicate removal
- memory filtering
- retrieval narrowing

Only the minimum required context should be transmitted.

## Current Implementation Status

The current iOS scaffold now includes a live OpenAI request path for Tier 3 escalation:

- uses `POST /v1/responses`
- sends a single compact PREXUS prompt instead of raw conversation replay
- uses user-supplied API keys from Keychain
- sets `store: false` for a stateless default
- extracts `output_text` content parts from the provider response
- falls back to local generation if the OpenAI request fails

This means OpenAI is no longer purely mocked in the runtime boundary, although Anthropic and Gemini remain placeholder integrations.

## Security Requirements

- User API keys must be user-supplied
- Keys must be stored in iOS Keychain
- Keys must never be hardcoded in the app or repo
- Transmission must respect PREXUS sensitivity policy
- Local-only memory items must never be forwarded

## Failure Handling

If OpenAI execution fails, PREXUS should:

- classify the failure type
- decide whether retry is appropriate
- offer fallback to local output where possible
- optionally reroute to another provider if user policy allows

## MVP Scope

For MVP, OpenAI integration should include:

- user API key configuration
- basic authenticated requests
- routing from PREXUS orchestrator
- compressed context packaging
- local fallback when cloud execution fails

MVP does not need:

- broad provider feature coverage
- complex background job orchestration
- advanced provider-specific optimizations beyond routing relevance
- streaming event handling on day one
